## 1. Executive Summary
This report details the systematic restoration of the LeadGen Engine's audio pipeline. The project was initially non-functional due to a combination of deprecated transport layers, serialization mismatches, and pipeline initialization failures. Through a process of deep-trace diagnostic logging and strategic refactoring, we restored full end-to-end voice-to-voice interaction.

---

## 2. Root Cause Analysis (RCA)

### Issue A: Pipeline Stalling (StartFrame Propagation)
- **Symptom:** The server would start, but no audio was ever processed or sent.
- **Root Cause:** A critical `StartFrame` was being blocked or swallowed by custom frame processors. In Pipecat, the `PipelineTask` will not transition to a `READY` state until the `StartFrame` has traversed the entire pipeline.
- **Solution:** Patched the `FrameProcessor` logic to ensure immutable propagation of control frames (`StartFrame`, `CancelFrame`) and refactored the pipeline to use standard components where possible.

### Issue B: Incompatible Transport Layer
- **Symptom:** `TypeError` and `ImportError` when attempting to use `WebsocketServerTransport`.
- **Root Cause:** The `WebsocketServerTransport` in the version of Pipecat used was either deprecated or required a different initialization pattern than what was in the boilerplate.
- **Solution:** Migrated to `FastAPIWebsocketTransport`. This allowed the application to share the same underlying `uvicorn` server as the static file server, significantly simplifying the networking stack and improving connection stability.

### Issue C: TTS Output Serialization Mismatch
- **Symptom:** Logs showed audio frames being generated, but the client received zero bytes.
- **Root Cause:** The `RawAudioSerializer` was strictly filtering for `OutputAudioRawFrame`. However, the newer `ElevenLabsHttpTTSService` produces `TTSAudioRawFrame` (a subclass).
- **Solution:** Broadened the serializer's induction logic to accept the base `AudioRawFrame` class, ensuring any binary audio generated by the synthesized speech service correctly reaches the WebSocket output.

### Issue D: Frontend Event Synchronization
- **Symptom:** Intermittent UI freezes and missed "Connected" status updates.
- **Root Cause:** A race condition in `index.html` where `getUserMedia` was requested before the WebSocket's `onopen` and `onmessage` handlers were fully registered.
- **Solution:** Reordered frontend initialization to ensure all event listeners are active before any media streams are opened.

## 3. Implementation Steps

### Step 1: Diagnostic Instrumentation
We introduced a custom `FrameLogger` to intercept and print every frame passing through the system. This revealed that audio was being generated (`TTSAudioRawFrame`) but discarded by the transport's serializer.

### Step 2: Service Refactoring
We switched from `ElevenLabsTTSService` (WebSocket-based) to `ElevenLabsHttpTTSService` (HTTP-based). This reduced complexity in the initial POC, as managing two simultaneous high-bandwidth WebSockets (one to the client, one to ElevenLabs) often led to timeout issues in limited environments.

### Step 3: Serializer Patching
We modified `server.py` to correctly handle the audio data:
```python
class RawAudioSerializer(FrameSerializer):
    async def serialize(self, frame: Frame) -> str | bytes | None:
        if isinstance(frame, AudioRawFrame): # Corrected from OutputAudioRawFrame
            return frame.audio
        return None
```

### Step 4: Automating the Greeting
To ensure immediate verification, we implemented a connection handler that automatically queues an AI greeting:
```python
@transport.event_handler("on_client_connected")
async def on_client_connected(transport, client):
    content = [{"role": "user", "content": "Hello! Please introduce yourself as Jessica..."}]
    await task.queue_frames([LLMMessagesFrame(content)])
```
